# Paper Analysis Skill

A skill for extracting knowledge elements from research paper PDFs and generating/registering notes based on the principles of Zettelkasten (Atomicity, Linking, Evolution).

---

## Basic Rules for PDF Analysis (Balancing Efficiency and Accuracy)

"When running this project, always use `dir_path="pdf"` as the default PDF location."

1. **Extraction Method Selection**: To balance speed and quality, execute in the following priority order:
   - **Preferred (Standard)**: High-quality extraction using `docling` (takes a few minutes)
     ```powershell
     python -m paper_memory extract "pdf/path.pdf"
     ```
     *Note: Extracts structured text and converts figures/tables to images. The extracted Markdown and images are automatically stored in the `extracted/<PDF Name>/` directory.*
   - **For High-Accuracy Table Extraction**: Run LLM-based table image analysis simultaneously
     ```powershell
     python -m paper_memory extract "pdf/path.pdf" --analyze-tables
     ```
   - **Fallback for Troubleshooting**: `pypdf` (text only)
     ```powershell
     python -m paper_memory extract "pdf/path.pdf" --use-pypdf
     ```
   - **Special (Explicit Request Only)**: `marker-pdf` (takes tens of minutes)
     ```powershell
     python -m paper_memory extract "pdf/path.pdf" --use-marker --light
     ```
     *Note: Only run this in the background with user permission if complex LaTeX mathematical formulas need to be extracted as text.*

2. **Text-First Analysis**: First, base your analysis on the Markdown generated by the above commands (or files under `extracted/`). Only refer to the images saved in `extracted/<PDF Name>/images/` if you need to visually check figures or tables.
   - **[IMPORTANT]** The `extracted/` directory is excluded by `.gitignore` and `.geminiignore`, making it invisible to standard file exploration tools. When reading Markdown files, specify the absolute path using the `view_file` tool or retrieve the file contents directly using PowerShell commands (e.g., `cat` or `Get-Content`).
3. **Phased Note Generation**: It is recommended to generate notes for a single paper over 3 turns:
   - Turn 1: `method`, `background`, `definition`
   - Turn 2: `result`, `discussion`
   - Turn 3: `conclusion`, `limitation`, `future_work`, `insight`

## Rules During Paper Analysis

### Atomicity

When extracting knowledge from a paper, you MUST classify each element into exactly one of the following `Type`s.
**[CRITICAL] You MUST output the EXACT string listed in the `Type` column (e.g., "background"). Do NOT output the `Description` or any other variations. If you output an invalid string, it will be automatically forced to "other".**

| Type | Description |
|------|-------------|
| `background` | Background / Prior research |
| `method` | Methodology / Approach |
| `result` | Results / Experimental data |
| `discussion` | Discussion / Interpretation |
| `conclusion` | Conclusion |
| `insight` | Author's insights / Inspiration |
| `limitation` | Limitations / Challenges |
| `future_work` | Future perspectives |
| `definition` | Important term definitions |
| `other` | Other (when difficult to classify) |

### Output Format

When knowledge is extracted from a paper, it MUST be output according to the JSON structure defined in `references/note-schema.json`.
**[CRITICAL] Do NOT output a simple array of notes.** You MUST output a single JSON object containing both the `"source_paper"` information and the `"notes"` array.
**[CRITICAL BILINGUAL REQUIREMENT]** For fields defined as bilingual dictionaries in the schema (such as `content`, `context`, `keywords`, `reason`), you MUST output BOTH the `"en"` key (English text) AND the `"local"` key (the user's preferred language, e.g., Japanese). Do NOT output a single string, and do NOT omit either language.

### Quality Standards

- Each note MUST be **independently understandable** (it should make sense without referring to other notes).
- **Thoroughness of Quantitative Data**:
  - If there are specific numerical values in the paper, record **as many of them as possible**.
  - When recording numerical values, **ALWAYS specify the unit** (e.g., GPU, Barrer, %, ℃, MPa).
  - For **separation membrane performance (permeability, selectivity, etc.)**, do not just record a single number; you MUST also include the **test conditions (e.g., CO2 concentration of feed gas, feed pressure, temperature, humidity)**.
- **Identification of Novelty and Core Concepts (The "Kimo")**:
  - Clearly describe "how this differs from previous papers (existing research)."
  - Specifically identify the "core" (key idea or ingenuity) of the technology or method.
- **Extraction of Verification and Challenges**:
  - Describe how the effectiveness of the proposed method was verified (comparison targets and evaluation metrics).
  - MUST extract any limitations, unresolved issues, or debatable points acknowledged by the authors themselves.
- `keywords` should include both **general terms and highly specialized terms** to maximize discoverability.
- `context` should describe "why this knowledge is important" and "in what situations it is useful."

## Note Registration Procedure

After extracting the knowledge, register it using the following steps:

```powershell
# 1. Save the extracted JSON to scratch/new_notes.json in UTF-8.
#    * Do NOT output the JSON directly in the chat response to prevent garbled text and save tokens; write it directly to the file.
# 2. Run the following command:
python -m paper_memory add --file scratch/new_notes.json --cleanup
# * Adding --cleanup automatically cleans up the scratch/ directory after processing.
# * If the source_paper's DOI is empty, it will be automatically populated via API.
```

## Link Generation Rules (A-Mem Evolution Principle)

After adding new notes, generate links using the following steps:

1. Once all notes have been registered, run bulk link construction using this command:
   ```powershell
   python -m paper_memory autolink --paper-title "Paper Title" --yes --quiet
   ```
2. After execution, confirm the summary (number of added links) and inform the user that detailed reasons can be checked in `logs/autolink_YYYYMMDD.log`.
3. Criteria for validating suggested links:
   - Deals with the same concept/method
   - Results are comparable
   - One serves as prerequisite knowledge for the other
   - Addressing similar challenges

## Reference Extraction Rules (Reading List)

Select cited literature that directly relates to the core findings/methods of the paper and register them in the Reading List.
If there is no applicable literature, you may skip this step (0 references is acceptable).

### Selection Criteria (ONLY if they match the following)

- Literature that forms the **direct foundation for the core methods or findings** of the paper.
- Prior research that the paper **directly compares against or improves upon**.
- Literature where the authors imply "this research would not be possible without this study."

### Exclusions

- Generic research background citations in the introduction.
- Standard citations for methodology (measurement techniques, analysis methods).
- Casual citations like "It has also been reported that..."

### Output Format

Output MUST follow the JSON structure defined in `references/reference-schema.json`.
**[CRITICAL BILINGUAL REQUIREMENT]** For fields like `reason` and `keywords`, you MUST output BOTH the `"en"` key AND the `"local"` key.

### Registration Procedure

1. If the extracted literature's DOI is explicitly stated in the text, include it; if unknown, leave it as an empty string `""` (Do NOT guess or hallucinate). **Web searches are NOT required.** Missing DOIs and bibliographic information will be automatically searched and populated by the backend Python script using the Crossref/OpenAlex APIs during registration.
2. Save the above JSON to `scratch/new_refs.json` in UTF-8.
3. Register using `python -m paper_memory refs-add --file scratch/new_refs.json --cleanup`
   - Duplicates will be automatically checked during registration (against existing reading lists and parsed notes).
